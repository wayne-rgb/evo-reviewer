# 测试运行策略

## 铁律：禁止随意跑全量测试

决策树：
1. 改了 1 个文件 → 只跑该文件对应的单个测试文件
2. 改了 1 个模块内的多个文件 → 只跑该模块的测试（unit 级）
3. 跨模块改动 / 多分支合并后 → 跑集成测试（cross 级）
4. 最终推送前，用户明确要求 → 跑全量测试

worktree / subagent 内部验证也只跑单文件测试，绝不跑全量。

## CI 策略

CI 由 evo-review 的 PostToolUse hook 驱动，git push 成功后自动开 subagent 后台执行，不阻塞对话。

流程：git push 成功 → hook 信号 → Claude 开 background subagent → 按改动范围选测试 → 结果反馈到对话

按改动范围选测试：
- 只改文档/配置 → 只跑治理门禁
- 单模块改动 → 该模块的 lint + 类型检查 + 单元测试
- 跨模块改动 → 上述 + 集成测试
- subagent 完成后汇报结果，失败时主动排查修复

## Bash 效率约束

1. **输出必须截断** — 可能产生大输出的命令必须加 `| tail -N`（默认 20 行）
2. **长命令交给 subagent** — 预期超过 2 分钟的命令（全量测试、大项目构建），交给 subagent 后台执行，主会话只收摘要
3. **不重复提交同一命令** — 失败了先分析原因，不盲目重试
4. **后台任务不堆积** — 同一时间最多 1-2 个后台命令
