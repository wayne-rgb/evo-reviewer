# 测试编写的 6 个维度

来源：全局 CLAUDE.md「测试编写的 6 个维度」

---

## 维度定义

写新测试时，覆盖所有**适用**的维度（不是每个测试都需要全部 6 个）：

| 编号 | 名称 | 定义 | 典型适用场景 |
|------|------|------|------------|
| 1 | **正常路径** | 核心功能成功执行 | 几乎所有测试都需要 |
| 2 | **副作用清理** | timer/lock/连接/文件句柄是否被正确清理 | 有状态的类/模块 |
| 3 | **并发安全** | 多线程/多协程/多设备并发操作同一资源 | 共享状态的单例/服务 |
| 4 | **错误恢复** | 错误处理路径被执行后，状态是否一致？ | 有 catch/finally 的代码 |
| 5 | **安全边界** | 超大/畸形/恶意输入 | 接受外部输入的接口 |
| 6 | **故障后可用性** | 故障后系统仍能处理后续请求 | 长生命周期的服务 |

---

## 适用性判断表

| 被测代码类型 | 必须覆盖 | 通常不需要 |
|-------------|---------|-----------|
| 纯函数/工具函数 | 1 | 2, 3, 5, 6 |
| 有状态的类（timer/lock/flag） | 1, 2, 4 | 3, 5 |
| 接受外部输入的接口（HTTP/WS/CLI） | 1, 4, 5 | 2, 6 |
| 单例服务（长生命周期） | 1, 2, 3, 4, 6 | 5 |
| 多设备/多用户共享的资源 | 全部 | 无 |

---

## 维度标注规范

写测试时用注释标注覆盖的维度，供 gate 统计维度分布、发现偏科：

| 语言 | 标注格式 | 位置 |
|------|---------|------|
| TypeScript | `// @dimension 1,3,6` | describe 块上方 |
| Go | `// @dimension 1,4,5` | Test 函数上方 |
| Swift | `// @dimension 1,2,4` | test 方法或 class 上方 |
| Python | `# @dimension 1,4` | test 函数上方 |

维度编号：1=正常路径 2=副作用清理 3=并发安全 4=错误恢复 5=安全边界 6=故障后可用性

**已有测试不强制补标注**，新测试必须标注。gate 的维度统计规则会自动检测未标注的新测试文件。

## dimension-coverage.yaml 格式说明

gate 在 Phase B 维度覆盖分析时，会读取项目根目录下的 `test-governance/dimension-coverage.yaml`，该文件记录每个测试文件覆盖的维度编号列表。

```yaml
# test-governance/dimension-coverage.yaml 格式
module-name:
  path/to/test-file.test.ts: [1, 3, 6]
  path/to/another.test.ts: [1, 4, 5]
```

**此文件由 /review Phase B 自动维护，开发者日常不需手动编辑。**

gate 的 `check_dimension_coverage` 函数从该文件读取数据，统计各维度的覆盖分布，发现偏科维度（某维度覆盖为 0）时输出警告。
