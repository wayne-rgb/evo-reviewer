# /test-check — 检查测试质量是否达标

快速检查指定测试文件的维度覆盖情况。只读不改。

维度定义：[dimensions.md](../review/references/dimensions.md)

## 输入

$ARGUMENTS — 测试文件路径或源码模块路径。

## 流程

1. 找到测试文件和被测源码，同时读取
2. 判断被测代码类型（纯函数 / 有状态类 / 外部接口 / 单例服务 / 共享资源）
3. 按维度定义的"适用性判断"表确定该代码需要哪些维度
4. 逐维度评估（搜索关键词判断覆盖情况）：

| 维度 | 判断方法 |
|------|---------|
| 1. 正常路径 | 有验证核心功能成功的用例 |
| 2. 副作用清理 | 有断言资源被清理（搜索：`null`、`clear`、`close`、`delete`、`cleanup`、`0`、`empty`） |
| 3. 并发安全 | 有多并发操作的用例（搜索：`并发`、`concurrent`、`parallel`、`race`，或语言特定的并发原语） |
| 4. 错误恢复 | 有注入失败的用例（搜索：`throw`、`error`、`fail`、`reject`、`fault`，或 mock 库的错误注入 API） |
| 5. 安全边界 | 有极端输入的用例（搜索：`超大`、`畸形`、`边界`、`limit`、`overflow`、`inject`、`malicious`） |
| 6. 故障后可用性 | 故障用例之后还有正常操作的断言（搜索：`后续`、`仍可用`、`still work`、`recover`） |

5. 输出评分卡：

```
## 测试质量评分卡：{文件名}

被测代码类型：{类型}
适用维度：{列出适用的维度编号}

| 维度 | 适用 | 状态 | 说明 |
|------|------|------|------|
| 1. 正常路径 | ✅ | ✅/❌ | ... |
| 2. 副作用清理 | ✅/— | ✅/❌ | ... |
| ... |

得分：X / Y（Y = 适用维度数）
```

如果需要补测试，引导用户使用 `/review`。
